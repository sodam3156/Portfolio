## 한눈에

- **제품 기획**: 사용자 문제, MVP 범위, 수용 조건, 서비스 청사진을 연결합니다.
- **AI 워크플로**: 역할별 에이전트, 구조화된 문서, 검토 게이트로 반복 업무를 설계합니다.
- **안전한 자동화**: 개인정보 최소수집, 로컬 우선, dry-run, 복구 가능한 변경을 기본값으로 둡니다.
- **실행 방식**: Python·GitHub Actions·HTML/CSS·Git을 활용해 문서에서 작동하는 결과물까지 연결합니다.

## 프로젝트 상태 기준

- `운영 중`: 실제 자동화가 반복 실행되는 프로젝트
- `구현됨`: 설치·실행·검증 가능한 코드가 있는 프로젝트
- `기획·프로토타이핑`: 사용자 흐름과 구현 명세를 만들고 MVP를 준비하는 프로젝트
- `운영 체계`: 제품을 만들기 위한 문서·프로세스 인프라

## 1. Syndy — 흩어진 일정 정보를 캘린더 행동으로

**상태:** 기획·MVP 설계

**역할:** 제품 기획·PM·사용자 흐름 설계

**키워드:** 일정 자동화 · 최소수집 · 사용자 확인

메신저, 이메일, 캡처 이미지에 흩어진 일정 후보를 추출하고 사용자가 확인한 뒤 캘린더에 등록하는 흐름을 설계했습니다. 플랫폼 메시지를 무리하게 수집하기보다 사용자가 제공한 입력을 처리하고 원문을 최소한으로 보관하는 방향을 택했습니다.

- PSSD 방식으로 문제·해결책·확장·개발 범위를 문서화
- MVP를 `입력 → 일정 후보 추출 → 사용자 확인 → 캘린더 등록`으로 축소
- 원문 즉시 폐기와 최소 데이터 저장을 핵심 설계 원칙으로 설정
- 팀 온보딩, 역할 분담, 기술·보안 검토 질문을 문서화

**현재 과제:** 실제 사용성 검증과 기술 스택 확정 전 단계이므로 성과나 운영 수치를 과장하지 않습니다.

## 2. 21세기 집현전 — 여러 관점을 관전하는 AI 토론 콘텐츠

**상태:** 세종 AX 해커톤 기획·프로토타이핑

**역할:** 기획/PM · 콘텐츠 계약 · 서비스 청사진 · 프론트엔드 요구사항

**키워드:** 멀티페르소나 · AI 토론 · 서비스 청사진 · 익명 세션

사용자가 생활 고민을 자유롭게 작성하고, 조선시대 콘셉트의 전문가 9명 중 약 4명을 선택해 서로 다른 관점의 토론을 관전하는 웹 콘텐츠입니다. 세종대왕은 정답을 선언하지 않고 의견 차이와 공통점을 정리하는 사회자로 동작합니다.

- `준비 → 실행 → 결과`의 서비스 청사진 작성
- 중앙 집현전 회의상과 세로형 단체채팅 피드로 토론 화면 설계
- 좌우 전문가 발언과 중앙 메시지 상태를 동기화하는 UI 명세 작성
- 개별 의견별 `공감해요 / 공감 어려워요` 피드백 설계
- 별점·점수·리더보드를 제외하고 생활 조건 비교에 집중
- React·FastAPI·LLM 병렬 호출·요약 체이닝·익명 세션을 후보 구조로 문서화

**현재 과제:** 구현 스택과 토론 생성 방식은 팀 킥오프에서 최종 확정할 예정입니다.

## 3. Founder File Organizer — 지우기 전에 증명하는 Windows 파일 정리 도구

**상태:** 구현됨 · 공개 저장소

**역할:** 제품 원칙·안전 요구사항·자동화 구조·검증

**키워드:** Python · SHA-256 · dry-run · Windows Automation

AI나 클라우드 서비스 없이 로컬에서 파일 분류, 정확한 중복 탐지, 정기 자동화를 수행하는 Windows 우선 도구입니다. 자동 정리보다 파일 손실 방지가 먼저라는 원칙으로 설계했습니다.

- 파일 크기와 SHA-256이 모두 같은 경우만 정확한 중복으로 판단
- 기본 동작은 미리보기이며, 명시적인 `--apply`에서만 변경
- 영구 삭제 대신 운영체제 휴지통 사용
- 0바이트 파일, symlink, reparse point, 개발 의존성 폴더 제외
- 로그인 시·지정 시각 실행을 위한 예약 작업 지원
- CI, CodeQL, Dependabot, 보안 신고 절차와 기여 가이드 구성

[공개 저장소 보기](https://github.com/sodam3156/Restruct-myPC)

## 4. Rulebook — 공고문을 AI가 읽을 수 있는 운영 데이터로

**상태:** 구현됨 · 공개 저장소

**역할:** 정보 구조·AI 작업 규칙·인덱스 자동화·개인정보 분리

**키워드:** Knowledge Architecture · Markdown · YAML · AI Workflow

정부지원사업, 공모전, 대외활동의 규정과 공고문을 일정한 폴더 구조로 저장하고 AI가 필요한 파일만 읽도록 만든 아카이브입니다.

- 공고마다 `meta.yaml`, `rules.md`, `notes.md`, `attachments/`를 분리
- 기계가 읽는 상태·마감 정보와 사람이 읽는 규정·전략을 구분
- 자동 생성 `INDEX.md`로 진행 상태와 마감 현황 제공
- `/add-program → /check-fit → /draft-application` 워크플로 설계
- 공개 규정 데이터와 비공개 프로필 정보를 저장소 수준에서 분리
- 개인정보를 공개 저장소에 복제하지 않는 운영 원칙 적용

[공개 저장소 보기](https://github.com/sodam3156/rulebook-public)

## 5. 고려대 세종캠퍼스 자동화 허브 — 학생 정보를 필요한 채널로

**상태:** 운영 자동화

**역할:** 자동화 기획·운영 구조·GitHub 워크플로 정비

**키워드:** Python · Discord · GitHub Actions · Crawling

공지, 학사일정, 셔틀버스, 대외활동, 식단 등 학생 생활 정보를 수집해 Discord 채널로 전달하는 봇 모음입니다.

- 여러 교내 정보원을 목적별 봇으로 분리
- 새 글이 있을 때만 알림을 전송해 중복과 소음을 줄임
- 학사일정을 월·분기·연도 단위로 조회하는 대화형 흐름 구성
- 매월 1일·15일에 필요한 일정만 자동 게시
- GitHub Actions 기반 스케줄과 상태 파일을 이용한 중복 방지
- 비밀값을 코드가 아닌 GitHub Secrets로 관리하는 구조로 정비

**공개 원칙:** 운영 저장소와 채널 정보는 비공개로 유지하고 기능 구조만 소개합니다.

## 6. AI Founder Workspace — 인간·AI 작업자를 위한 실행 운영체계

**상태:** 운영 체계 설계

**역할:** 제품 운영 모델·작업 계약·검토 구조 설계

**키워드:** Human-AI Collaboration · Task Contract · Independent QA

아이디어를 제품 가설, 검증 가능한 백로그, MVP로 연결하기 위한 문서 중심 작업 공간입니다. 많은 에이전트를 만드는 것보다 책임·검토·승인 기준을 명확히 하는 데 집중합니다.

- 제품 요약, 백로그, 결정 기록을 단일 기준 문서로 분리
- `한 작업 = 한 책임자 = 한 브랜치/워크트리` 원칙
- PM Dispatcher, Engineering, Independent QA의 최소 역할 구조
- 결과·범위·수용 조건·예산·롤백을 담는 Task Contract
- 읽기·초안·내부 변경·외부 변경의 권한 등급
- 비용·시간·재시도 상한과 독립 검토 게이트

**공개 원칙:** 개인 PC 진단, 로컬 경로, 계정 정보가 포함된 운영 기록은 공개하지 않고 재사용 가능한 방법론만 소개합니다.

## 준비 중 프로젝트 — 함께 만들 사람을 찾습니다

### NAVIGATOR — 소비 기록을 다음 행동으로 바꾸는 금융 학습 도구

**상태:** 문제·해결 가설 검증 중

계좌를 연결하지 않고 거래 내역 파일을 직접 올리면 소비 흐름을 이해하기 쉽게 정리하고, 한 달 동안 실천할 행동 계획으로 바꾸는 서비스입니다.

- **검증할 가설:** 기록만 남기는 가계부보다 내 데이터에 맞는 다음 행동을 원한다.
- **다음 검증:** 파일 업로드 의향, 결과 이해도, 행동 계획의 실행 가능성 인터뷰
- **함께할 역할:** 데이터·백엔드, 금융 도메인 검토, 사용자 리서치
- **주의:** 금융상품 추천이나 수익을 약속하지 않으며 보안·규제 검토가 필요합니다.

### TrustLens — 리뷰를 다시 확인하는 시간을 줄이는 브라우저 도구

**상태:** 문제·비용 가설 검증 중

리뷰 페이지에서 광고·반복 표현의 신호와 핵심 장단점을 근거와 함께 보여주는 도구입니다. 규칙 기반 분석과 가벼운 분류 모델을 우선하고 필요한 부분에만 LLM을 사용합니다.

- **검증할 가설:** 완벽한 진위 판정보다 빠르게 확인할 위험 신호와 근거를 원한다.
- **다음 검증:** 리뷰 교차 확인 시간과 신뢰 판단에 필요한 신호 인터뷰
- **함께할 역할:** 브라우저 확장 프론트엔드, 데이터·ML, 리서치·실험 설계
- **주의:** 정확도·비용 절감·수익성은 아직 검증되지 않았습니다.

## 작업에서 반복되는 원칙

1. **문제부터 좁힙니다.** 기능 목록보다 사용자의 한 번의 성공 행동을 먼저 정의합니다.
2. **계획과 구현을 구분합니다.** 실행되지 않은 내용은 성과처럼 쓰지 않습니다.
3. **복구 가능성을 기본값으로 둡니다.** dry-run, 백업, 검증, 롤백을 변경보다 먼저 설계합니다.
4. **개인정보를 구조적으로 줄입니다.** 수집하지 않는 것이 가장 강한 보호라고 봅니다.
5. **문서와 코드가 함께 움직이게 합니다.** 수용 조건과 검증 결과를 저장소에 남깁니다.
